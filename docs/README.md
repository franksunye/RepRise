# 设计问题

## 三个业务内容，实际的通话分析、成交数据以及即时奖励活动。

- 展示已经存在的，真实的“通话分析”，这属于哪个模块？（最终我们必将靠近营收Revenue），这个真实的对话Conversation与模拟的关系是什么？

- 管家（销售）的视图，也会包括实战的表现，即签约的数据，与当前的模块之间的关系，如何产品化这个部分的内容？

- 奖励部分，是不是也应该归入到“赋能系统”中，这本身是实战数据

## 设计上的挑战，分散的数据，多渠道的沟通，通用化和产品化的目标。

- 如何做到产品化，分散的数据不会影响良好的架构和系统。销售赋能系统主要包括培训、教练、外延到营收分析、销售实际能力分析、销售激励
- 已经有了Java的后台架构、MongoDB的数据库、Metabase的分析系统，是一套整合了CRM，现场服务和后台调度、结算等的总和运营平台，如果我们的销售赋能系统选用Vercel部署、Next.js作为主要的技术栈
